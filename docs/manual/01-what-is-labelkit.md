# 第 1 章　LabelKit 是什么

> 本章告诉你 LabelKit 解决什么问题、它如何思考数据加工这件事，以及它刻意不做什么。
> 读完本章，你应当能用自己的话向同事解释「LabelKit 是一条什么样的流水线」。

## 1.1 它解决的问题

想象你手里有一大袋刚从各处采集来的数据——可能是几万条用户在输入法里敲下的请求，也可能是几千组手机屏幕的截图。这袋数据是「生的」：

- 里面有**大量重复**——同一句话被采集了三遍，只差一个空格；
- 里面有**垃圾**——"哈哈哈哈哈"、乱码、空洞的寒暄；
- 它们**没有标签**——你不知道哪条是翻译请求、哪条是写作请求，不知道哪张截图是登录页、哪张是设置页；
- 而下游（模型训练、评测集建设、数据入库）需要的是**干净、有标签、结构统一**的数据。

传统做法是雇标注员：贵、慢、标准还因人而异。LabelKit 的做法是**把这套加工流程固化成一条可配置的流水线，让大语言模型（LLM）来当质检员和标注员，让代码来当监工**：

```
原始数据 ──▶ （可选）分段 ──▶ （可选）缝合 ──▶ 去重 ──▶ （可选）分类 ──▶ （可选）动作摘取 ──▶ 质量打分 ──▶ 自动标注 ──▶ （可选）二次校验 ──▶ 结构合法的 JSONL
                  │                             │                                             │             │                │
                  └─────────────────────────────┴─────────────────────────────────────────────┴──── 被淘汰的记录进「拒绝通道」，一条不丢、笔笔有账 ────┘
```

一句话定义：**LabelKit 是一个基于 LLM API 的无状态批处理命令行工具**。你给它一个输入路径、两份配置文件，它给你一个结构由你定义、由代码引擎保证合法的 JSONL 输出文件，外加一份运行报告。

从 v1.4 起它还多了一种玩法：**纯生成模式**——手里一条数据都没有时，让它从几条种子例句出发，从零合成一个数据集，合成品照样走完整套质检流程（见第 12 章）。

## 1.2 一个贯穿全书的例子

本手册会反复使用同一个例子，让你在每一章都能落地想象：

> 你在维护一个中文输入法。输入法每天采集到大量用户敲给 AI 助手的一句话请求，例如
> 「帮我写一条请假条，明天上午要去医院」。你想把这批数据加工成一个意图分类数据集：
> 每条请求标注出**意图类别**（写作协助 / 问答 / 翻译 / 闲聊 / 其他）、**主题**和**难度**。

用 LabelKit 做这件事，你要提供的全部东西是：

1. 一个 JSONL 文件（每行一条 JSON，里面有 `instruction` 字段装着那句话）；
2. 一份 `config.toml`——告诉工具「LLM 服务在哪、密钥叫什么名字」；
3. 一份 `project.toml`——告诉工具「输入在哪、输出到哪、要不要去重、质量线画在哪、
   标注指令怎么说、输出的 JSON 长什么样」。

然后运行一条命令：

```bash
labelkit run --config config.toml --project project.toml
```

几分钟后你会得到：

- `out/text-labels.jsonl` —— 主输出，每行都是 `{"intent": ..., "topic": ..., "difficulty": ..., "_meta": {...}}`——你声明的三个字段**保证**符合你的 JSON Schema；`_meta` 是工具默认随行附加的元信息（质量分、溯源、标注痕迹，详见第 8 章），不想内嵌可设 `output.meta_mode = "sidecar"` 或事后用 `jq 'del(._meta)'` 剥除；
- `out/text-labels.rejects.jsonl` —— 被淘汰记录的「案底」：谁被淘汰了、在哪个环节、为什么；
- `out/text-labels.report.json` —— 运行报告：各环节计数、质量分分布、token 用量（给模型档配置单价后还会附成本估算，见第 6、8 章）。

这就是 LabelKit 的全部交互面：**命令行 + 两份 TOML + 输入输出文件**。没有服务进程，没有数据库，没有网页界面。

## 1.3 五个核心设计信念

理解这五条，后面所有章节的设计都会显得顺理成章。

**① 无状态：跑完即忘。**
LabelKit 像一台绞肉机而不是一座仓库：数据从一头进、另一头出，机器本身不存肉。所有中间状态（去重索引、质量分、未定稿的标注）只活在进程内存里，进程退出即销毁。它不建缓存、不写临时文件、不支持断点续跑。这换来的是极简的心智模型：**每次运行都是干净的、完整的、可复现的**。

**② LLM 的输出不可信。**
LLM 是个才华横溢但不太守规矩的实习生：它可能在 JSON 外面套一层 Markdown 围栏，可能多写一个逗号，可能把 `writing_assist` 写成 `writing`。所以 LabelKit 立了一条铁律：**凡是 LLM 说的，必须经过代码校验才能进入输出**。这就是第 14 章「结构引擎」的四层防线。主输出里的每一行都必然通过了你的 JSON Schema 校验——这不是「大概率」，是机制保证。

**③ 记录级隔离：一条坏数据不能拖垮一锅汤。**
某条记录标注失败、某张图解码失败，只影响它自己——它被标记为失败、进拒绝通道、计入报告，运行继续。你永远不会因为第 47231 条数据有问题而废掉一整晚的运行。

**④ 一切可审计。**
每条被淘汰的记录都有去向和理由；每个质量分都能在 trace 日志里找到 LLM 当时的裁决和理由；报告里的计数满足严格的守恒等式（进 = 出 + 各类淘汰）。你不需要「相信」LabelKit，你可以查账。

**⑤ 配置即契约。**
两份 TOML 在启动时被**全量校验**——所有错误一次性报完（而不是改一个错跑一次发现下一个），校验通过后配置被冻结，运行期不会再冒出配置错误。写错的枚举值、引用了不存在的模型 profile、非法的 JSON Schema，都活不过启动那一秒。

## 1.4 流水线上有哪些工位（算子总览）

LabelKit 把每个加工环节叫做一个**算子**（Stage）。它们像流水线上的工位，每个都可以独立开关：

| 算子 | 做什么 | 默认 | 打个比方 |
|---|---|---|---|
| **ingest 接入** | 读入数据、构造记录、剔除坏行 | 恒开 | 收货口：验货、贴上唯一编号 |
| **segment 分段** | 把时间序的记录流切成一个个语义完整的操作片段（episode），并剔除噪声帧（v1.8，详见第 25 章） | 关 | 剪辑台：把整卷录像剪成一段段完整的「一件事」，废镜头丢掉 |
| **stitch 缝合** | 把同一任务被穿插切开的片段保守缝合成完整线索（thread），过短被误剔的收尾帧可救援（v1.9，详见第 26 章） | 关 | 装订台：被打断的连载稿件按线索装订成册，掉落的散页拾回原册 |
| **dedup 去重** | 精确 + 近似去重（文本指纹 / 图像指纹） | 开 | 安检仪：同样的包裹只放一件过去 |
| **classify 分类** | 用 LLM 把数据按你的类别表分门别类，后续工位可按类改换参数（v1.7，详见第 24 章） | 关 | 分拣台：包裹先按品类分堆，后面的工位对不同的堆换不同做法 |
| **extract 摘取** | 对 episode 的每对相邻帧推断发生了什么动作，产出结构化动作序列（v1.8，详见第 25 章） | 关 | 解说员：看前后两帧，说出中间发生的那一步操作 |
| **quality 质量打分** | 用 LLM 按评价准则（rubric）打分，可按线淘汰 | 开 | 质检员：给每件货打分，不达标的下线 |
| **annotate 标注** | 用 LLM 按你的指令生成符合 Schema 的标注 | 开 | 贴标员：按你的模板给货贴标签 |
| **generate 生成** | 以高质量记录为种子，让 LLM 合成新样本 | 关 | 复制车间：照着样品仿制新货（仿品也要重新过安检和质检） |
| **verify 二次校验** | 用另一个 LLM 当审核员复查标注，可驱动返工 | 关 | 抽检科：独立复核，不合格的打回重做或下线 |
| **emit 输出** | 三通道落盘：主输出 / 拒绝 / 报告 | 恒开 | 发货口：良品装箱，次品留档，账目归档 |

两个「恒开」的工位（接入、输出）之外，中间九个都由 `project.toml` 里各自的 `enabled` 开关控制。但组合不是完全自由的——比如 `verify` 依赖 `annotate`（没有标注就没有可校验的东西），`quality` 和 `annotate` 至少要开一个（否则这次运行没有产出意义）。合法组合与约束详见第 4 章。

数据在工位之间以**批**（batch，默认 256 条）为单位流动。批还有一个特殊身份：它是质量打分做「成对比较」时的比较池——这一点在第 10 章会变得非常重要。

## 1.5 两种数据形态（模态）

LabelKit 原生支持两类输入，配置里叫 `modality`：

- **`text` 文本模态**：一个 JSONL 文件（或装满 JSONL 的目录），每行一条 JSON，由你指定其中哪个字段是「正文」。
- **`ui` 屏幕模态**：一个目录，里面成对地放着屏幕截图（`image_7.png`）和 UI 控件树文件（`uitree_7.jsonl`），LabelKit 按文件名里的编号自动配对，允许两个文件分属不同子目录。标注时截图和控件树会一起喂给视觉模型。

v1.8 起，两种模态的输入还可以带上**时间轴**：连续采集的记录流（如一次录屏的逐帧截图序列）开启分段算子后会先被切成一个个语义完整的操作片段（episode），再以片段为单位走后续工位（第 25 章）。

数据应当如何准备与排布，是第 5 章的主题。

## 1.6 它刻意不做什么

工具的边界和能力同样重要。LabelKit **不做**：

- **不训练模型**——所有打分、标注都通过 API 调用现成的 LLM 完成；
- **不存数据**——没有数据库、没有缓存、没有断点续跑（这是特性，不是缺陷）；
- **不做人工标注界面**——需要人工复核时，把输出导入 Argilla 之类的平台；
- **不做数据采集和版本管理**——它只负责「加工」这一段；
- **不发任何多余的网络请求**——数据只去配置里声明的 LLM 端点，无遥测、无更新检查；
- **不提供服务化形态**——它是一次性命令，不是常驻服务。

规模上限的设计目标是**单次运行 ≤ 50 万条记录**（默认配置约占 2–4 GB 内存）。更大的量，按目录切分成多次运行即可。

## 1.7 算法有出处

LabelKit 里每个算法环节都不是拍脑袋来的：质量打分的成对比较 + Bradley-Terry 拟合来自 QuRating（ICML 2024），单点加性打分来自 FineWeb-Edu（NeurIPS 2024），近似去重的 MinHash-LSH 是 LLM 训练数据处理的方法学标准（Lee et al., ACL 2022），数据生成是 Self-Instruct（ACL 2023）流程，二次校验是 LLM-as-a-Judge（NeurIPS 2023）与 Self-Refine 的组合。你不需要读这些论文也能用好工具，但当你想深究「为什么这么设计」时，设计规格书（`spec/` 目录）里每一处都有引注。

## 1.8 本手册的阅读路线

- **第一次接触**：顺读第 2、3 章，20 分钟内跑通第一个例子。
- **准备正式使用**：读第 4～8 章（概念、数据、配置、输出），这是使用者的必修课。
- **想把效果调到最好**：第 9～14 章逐算子精讲，每章都有「调优」小节；第 16 章教你用 trace 日志驱动评价准则迭代。
- **动手抄作业**：第 19～23 章是五个从易到难的完整实战教程，全部基于真实可运行的示例工程。
- **随手查**：附录 A 是全参数速查表；第 18 章是错误码与故障排查；附录 B 是默认评价准则全文。
