# 第 20 章　教程二：质量门实战——画线、看账、调准则

> **难度：★★☆☆☆**
> 舞台是仓库自带的 `examples/text`（14 条故意掺了脏数据的输入法请求）。这个工程把纯文本
> 格式能开的算子全开了（分类、生成、评审都在），但本章只盯**质量门**这一个工位；
> 目标：学会一套「先打分后画线」的科学流程，并体验一次 rubric 迭代。
> 本章所有数字都来自真实运行；你照跑账目会与此接近，但 LLM 打分有非确定性，
> 阈值附近的一两条边界记录可能进出浮动（守恒等式恒成立）。

## 20.1 起点：跑一遍看账

```bash
cd examples/text && mkdir -p out
set -a && source ../../.env && set +a
uv run labelkit run --config ../config.toml --project project.toml
```

工程配置的质量节是 `mode="pointwise"`、`rubric="default:text"`，全局 `threshold=0.25`，另有两条**按类覆盖**（v1.7 的分类算子在本工程是开的）：writing 类放宽到 0.2、qa 类收严到 0.4。跑出来的账：

```
scanned=14  generated=12  dropped_dup=1  dropped_lowq=10  emitted=15
```

**质量门总共干掉 10 条**：输入侧 14 条里去重先拦 1 条、质量门拦 4 条；另外 12 条生成样本回流后被**同一把尺子**刷掉 6 条（生成算子以过门记录为种子，见第 12 章——本章只看门本身）。输入侧 13 条活 9 条、合成侧 12 条活 6 条——多还是少？别拍脑袋，看两份材料。

## 20.2 材料一：直方图与准则均值（report.json）

```json
"aggregate_histogram": {"0.0-0.1": 3, "0.1-0.2": 3, "0.2-0.3": 6, "0.3-0.4": 5,
                         "0.4-0.5": 2, "0.5-0.6": 4, "0.6-0.7": 1, "0.7-0.8": 1,
                         "0.8-0.9": 0, "0.9-1.0": 0},
"per_criterion_mean": {"educational_value": 0.368, "facts_trivia": 0.160,
                        "required_expertise": 0.288, "writing_style": 0.400}
```

读出三件事：

1. 25 条被打分的记录（13 输入 + 12 合成）有 **11 条挤在 0.2~0.4**——分布的重心正好压在三条门线（0.2 / 0.25 / 0.4）附近。质量线切在人堆里意味着：**阈值动一格、或打分漂一格，都会成批改变去留**——这正是「阈值敏感区」的形状；
2. `facts_trivia` 均值 0.16，四条准则里最低——这批以闲聊、写作协助为主的数据在「事实与知识含量」上得分普遍偏低，这条准则在**系统性压低**聚合分；
3. 分类开启时报告里还有 `quality.by_class`——每个类池一套直方图与均值（writing 池 9 条、qa 池 7 条、translation 池 6 条、other 池 3 条）。**按类画线就要按类看图**：qa 池的分布明显靠右（知识型请求得分高），0.4 的严线是有依据的；writing 池整体贴地，0.2 的宽线也是有依据的。

## 20.3 材料二：淘汰名单（rejects）

```bash
jq -r '._meta | "\(.source.line_no)\t\(.reason)\t\(.label)"' out/text-labels.rejects.jsonl | sort -n
```

对回输入文件逐行看（`label` 列是类标签；`line_no` 为空的是合成样本）：输入侧被质量门淘汰的四条里有「哈哈哈哈哈哈哈」「在吗」「随便聊聊」（该杀，全在 other 类、门线 0.25），但也有**一条完全正常的请假条请求**（第 1 行，writing 类）！更扎眼的是它的孪生兄弟：第 7 行那条只多一个「做」字的近似改写**活了下来**——聚合分恰好 0.2，贴着 writing 类已经放宽到 0.2 的门线幸存（门控规则是 `< threshold` 才淘汰）。两条几乎一样的文本一死一生——**日常写作类请求整体被这把尺子按在了生死线边缘**，措辞浮动一下就换个结局。

诊断：`default:text` 的口径是「**作为训练语料的价值**」——知识密度、专业度权重很高。而这个工程的真实目标是「**保留多样的真实用户意图**」——寒暄该杀，但请假条、周报这类日常请求恰恰是最有代表性的意图样本，不该在生死线上挣扎（按类放宽到 0.2 只是缓解，没有换口径）。**不是阈值错了，是尺子不对。**

## 20.4 迭代：换一把对口径的尺子

方法照第 15.5 节的流程：改配置 → `--limit` 小样本验证 → 对比。做实验时先把变量缩到质量门本身：临时置 `generate.enabled = false`（不然每轮重跑都会连带一批新的合成样本），并删掉两条 `[class.*.quality]` 按类覆盖（旧尺子上画的线对新尺子没有意义）。然后把 rubric 换成自定义两准则（口径：意图是否真实明确，而非知识含量）：

```toml
[quality]
enabled = true
mode = "pointwise"
llm = "default"
threshold = 0.4
rubric = "inline"

[rubric]
name = "ime-intent-v1"        # 内联 rubric 必填的标识（缺了它校验直接退出码 2）

[[rubric.criteria]]
key = "intent_clarity"
weight = 2.0
description = "意图明确度：这条请求是否表达了一个明确、可执行的诉求。"
pairwise_prompt = "比较两条请求，哪一条的诉求更明确、更可直接执行？"
pointwise_levels = [
  "0: 无任何可辨认的诉求（纯语气词、乱码、单字）。",
  "1: 隐约有方向但无法执行（如'在吗'）。",
  "2: 在 1 的基础上，诉求可辨认但缺关键信息。",
  "3: 在 2 的基础上，诉求明确、可直接执行。",
  "4: 在 3 的基础上，带具体约束（时间/对象/格式）。",
  "5: 在 4 的基础上，约束完备，堪称模范请求。"]

[[rubric.criteria]]
key = "authenticity"
weight = 1.0
description = "真实性：是否像真实用户会敲出的自然请求（而非测试串或无意义输入）。"
pairwise_prompt = "比较两条请求，哪一条更像真实用户的自然输入？"
pointwise_levels = [
  "0: 明显是键盘噪声或测试输入。",
  "1: 勉强像句话但极不自然。",
  "2: 在 1 的基础上，基本自然但空洞。",
  "3: 在 2 的基础上，自然且有真实场景感。",
  "4: 在 3 的基础上，场景具体可信。",
  "5: 在 4 的基础上，典型到可以进用户研究案例库。"]
```

重跑后你会看到请假条、翻译这类请求回到 0.6+ 档（意图明确、真实），「哈哈哈哈」「在吗」仍在 0.2 以下——输入侧 13 条**留下 10~11 条、淘汰 2~3 条**，这才是这个任务想要的口径。

> 通用心法：**淘汰名单里出现「你明明想要的数据」，第一反应查 rubric 口径，第二反应才是调阈值。**阈值解决「切多深」，rubric 解决「按什么切」。

## 20.5 变奏一：只打分不过滤，下游后筛

拿不准就先不切——删掉 `threshold`：

```toml
[quality]
mode = "pointwise"
rubric = "inline"    # 上面那套
# 不设 threshold ⇒ 只打分，全部放行
```

所有记录（连「哈哈哈哈」）都会带着分数进主输出，然后：

```bash
jq -c 'select(._meta.scores["__aggregate__"] >= 0.4)' out/text-labels.jsonl | wc -l
```

**门控留宽 + 下游后筛**是探索期的推荐姿势：当次没淘汰的随时能再筛，当次淘汰掉的只能重跑找回（重新花钱）。

## 20.6 变奏二：pairwise + top_ratio（定量保留）

需求换成「每批只留最好的一半」时，切到锦标赛模式：

```toml
[quality]
mode = "pairwise"
rounds = 4
selection = "top_ratio"
top_ratio = 0.5            # 保留批内前 50%（此时不得再设 threshold）
rubric = "inline"
```

两个必须复述的语义（第 10 章）：pairwise 的分数是**批内百分位**（每批最低 0、最高 1，跨批不可比，均值天然 ≈0.5）；top_ratio 按**名次**保留，与百分位天然自洽。成本上不吃亏：2N 次调用 vs pointwise 双准则的 2N，正好持平——pairwise 的调用数与准则条数无关，准则达到 3 条及以上时它开始更便宜（见 10.9 成本速算表）。

## 20.7 变奏三：给裁决留理由，算一次 tie 率

调优期在 project.toml 打开（示例工程已开，通道更全）：

```toml
[trace]
enabled = true
channels = ["classify", "quality", "verify", "schema"]
content = "refs"
```

pointwise 模式下每次打分的 `reason` 都在 trace 里；pairwise 下按第 16 章的 jq 算各准则 tie 率——高 tie 率准则就是你 rubric 里的「废话准则」，改写或删掉。

## 20.8 本教程的可迁移结论

1. **画线三步**：先无阈值跑小样本 → 读直方图定切点 → 读 rejects 验证切掉的确实是不想要的；
2. **均值明显偏低的准则**（如这里的 facts_trivia ≈0.16，四条准则里最低）要么不适用于你的数据，要么权重该降——它在无差别地拉低所有人；
3. rubric 口径 > 阈值数值：**先对尺子，再定深浅**；
4. pointwise = 绝对刻度可后筛，pairwise = 批内相对可定量，选型看下游怎么用分数。
