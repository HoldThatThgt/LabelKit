# 第 20 章　教程二：质量门实战——画线、看账、调准则

> **难度：★★☆☆☆**
> 舞台是仓库自带的 `examples/text`（14 条故意掺了脏数据的输入法请求）。
> 目标：学会一套「先打分后画线」的科学流程，并体验一次 rubric 迭代。
> 本章所有数字都来自真实运行；你照跑账目会与此接近，但 LLM 打分有非确定性，
> 阈值附近的一两条边界记录可能进出浮动（守恒等式恒成立）。

## 20.1 起点：跑一遍看账

```bash
cd examples/text && mkdir -p out
set -a && source ../../.env && set +a
uv run labelkit run --config ../config.toml --project project.toml
```

工程配置的质量节（`mode="pointwise"`、`threshold=0.25`、`rubric="default:text"`）跑出来的账：

```
scanned=14  dropped_dup=1  dropped_lowq=5  emitted=8
```

**14 条活下来 8 条。**去重干掉 1 条，质量门干掉了 5 条——多还是少？别拍脑袋，看两份材料。

## 20.2 材料一：直方图与准则均值（report.json）

```json
"aggregate_histogram": {"0.0-0.1": 3, "0.1-0.2": 2, "0.2-0.3": 2, "0.3-0.4": 0,
                         "0.4-0.5": 3, "0.5-0.6": 0, "0.6-0.7": 2, "0.7-0.8": 1,
                         "0.8-0.9": 0, "0.9-1.0": 0},
"per_criterion_mean": {"educational_value": 0.308, "facts_trivia": 0.215,
                        "required_expertise": 0.323, "writing_style": 0.400}
```

读出三件事：

1. 分布**双峰**：一堆记录挤在 0~0.3（垃圾与日常请求），另一堆在 0.4+（知识型与带约束的请求），0.3-0.4 之间是空的——0.25 的线切在左峰内部，把左峰的大部分切掉了；
2. `facts_trivia` 均值 0.215，四条准则里最低——这批以闲聊、写作协助为主的数据在「事实与知识含量」上得分普遍偏低，这条准则在**系统性压低**聚合分；
3. 空档 0.3-0.4 提示：线画 0.3 或 0.35 结果完全一样（都是切掉左峰）——**阈值的敏感区间在峰内，不在峰间**。

## 20.3 材料二：淘汰名单（rejects）

```bash
jq -r '._meta | "\(.source.line_no)\t\(.reason)"' out/text-labels.rejects.jsonl | sort -n
```

对回输入文件逐行看：被质量门淘汰的五条里有「哈哈哈哈哈哈哈」「在吗」「随便聊聊」（该杀），但也有**两条完全正常的请假条请求**（第 1、7 行）！再看压线的：翻译请求（第 2 行）是以聚合分恰好 0.25 贴线幸存的（门控规则是 `< threshold` 才淘汰）——**日常写作/翻译类请求整体被这把尺子按在了生死线边缘**，哪边浮动一下就掉下去。

诊断：`default:text` 的口径是「**作为训练语料的价值**」——知识密度、专业度权重很高。而这个工程的真实目标是「**保留多样的真实用户意图**」——寒暄该杀，但请假条、周报这类日常请求恰恰是最有代表性的意图样本，不该在生死线上挣扎。**不是阈值错了，是尺子不对。**

## 20.4 迭代：换一把对口径的尺子

方法照第 15.5 节的流程：改配置 → `--limit` 小样本验证 → 对比。把 rubric 换成自定义两准则（口径：意图是否真实明确，而非知识含量）：

```toml
[quality]
enabled = true
mode = "pointwise"
llm = "default"
threshold = 0.4
rubric = "inline"

[rubric]
name = "ime-intent-v1"        # 内联 rubric 必填的标识（缺了它校验直接退出码 2）

[[rubric.criteria]]
key = "intent_clarity"
weight = 2.0
description = "意图明确度：这条请求是否表达了一个明确、可执行的诉求。"
pairwise_prompt = "比较两条请求，哪一条的诉求更明确、更可直接执行？"
pointwise_levels = [
  "0: 无任何可辨认的诉求（纯语气词、乱码、单字）。",
  "1: 隐约有方向但无法执行（如'在吗'）。",
  "2: 在 1 的基础上，诉求可辨认但缺关键信息。",
  "3: 在 2 的基础上，诉求明确、可直接执行。",
  "4: 在 3 的基础上，带具体约束（时间/对象/格式）。",
  "5: 在 4 的基础上，约束完备，堪称模范请求。"]

[[rubric.criteria]]
key = "authenticity"
weight = 1.0
description = "真实性：是否像真实用户会敲出的自然请求（而非测试串或无意义输入）。"
pairwise_prompt = "比较两条请求，哪一条更像真实用户的自然输入？"
pointwise_levels = [
  "0: 明显是键盘噪声或测试输入。",
  "1: 勉强像句话但极不自然。",
  "2: 在 1 的基础上，基本自然但空洞。",
  "3: 在 2 的基础上，自然且有真实场景感。",
  "4: 在 3 的基础上，场景具体可信。",
  "5: 在 4 的基础上，典型到可以进用户研究案例库。"]
```

重跑后你会看到请假条、翻译这类请求回到 0.6+ 档（意图明确、真实），「哈哈哈哈」「在吗」仍在 0.2 以下——**留下 11~12 条、淘汰 2~3 条**，这才是这个任务想要的口径。

> 通用心法：**淘汰名单里出现「你明明想要的数据」，第一反应查 rubric 口径，第二反应才是调阈值。**阈值解决「切多深」，rubric 解决「按什么切」。

## 20.5 变奏一：只打分不过滤，下游后筛

拿不准就先不切——删掉 `threshold`：

```toml
[quality]
mode = "pointwise"
rubric = "inline"    # 上面那套
# 不设 threshold ⇒ 只打分，全部放行
```

所有记录（连「哈哈哈哈」）都会带着分数进主输出，然后：

```bash
jq -c 'select(._meta.scores["__aggregate__"] >= 0.4)' out/text-labels.jsonl | wc -l
```

**门控留宽 + 下游后筛**是探索期的推荐姿势：当次没淘汰的随时能再筛，当次淘汰掉的只能重跑找回（重新花钱）。

## 20.6 变奏二：pairwise + top_ratio（定量保留）

需求换成「每批只留最好的一半」时，切到锦标赛模式：

```toml
[quality]
mode = "pairwise"
rounds = 4
selection = "top_ratio"
top_ratio = 0.5            # 保留批内前 50%（此时不得再设 threshold）
rubric = "inline"
```

两个必须复述的语义（第 10 章）：pairwise 的分数是**批内百分位**（每批最低 0、最高 1，跨批不可比，均值天然 ≈0.5）；top_ratio 按**名次**保留，与百分位天然自洽。成本上不吃亏：2N 次调用 vs pointwise 双准则的 2N，正好持平——pairwise 的调用数与准则条数无关，准则达到 3 条及以上时它开始更便宜（见 10.9 成本速算表）。

## 20.7 变奏三：给裁决留理由，算一次 tie 率

调优期在 project.toml 打开（示例工程已开）：

```toml
[trace]
enabled = true
channels = ["quality", "schema"]
content = "refs"
```

pointwise 模式下每次打分的 `reason` 都在 trace 里；pairwise 下按第 16 章的 jq 算各准则 tie 率——高 tie 率准则就是你 rubric 里的「废话准则」，改写或删掉。

## 20.8 本教程的可迁移结论

1. **画线三步**：先无阈值跑小样本 → 读直方图定切点 → 读 rejects 验证切掉的确实是不想要的；
2. **均值明显偏低的准则**（如这里的 facts_trivia ≈0.22，四条准则里最低）要么不适用于你的数据，要么权重该降——它在无差别地拉低所有人；
3. rubric 口径 > 阈值数值：**先对尺子，再定深浅**；
4. pointwise = 绝对刻度可后筛，pairwise = 批内相对可定量，选型看下游怎么用分数。
