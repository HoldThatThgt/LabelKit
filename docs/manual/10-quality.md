# 第 10 章　质量算子 quality：给数据开锦标赛

> 这是流水线上最烧钱、也最值得吃透的算子。读完本章你应当能回答三个问题：
> **pairwise 和 pointwise 该选哪个？threshold 画在哪才合理？rubric 怎么写才有区分度？**

## 10.1 直觉：质检员手里的两种量具

给数据打质量分，本质是让 LLM 当质检员。LabelKit 给质检员配了两种量具：

- **pointwise（单点打分）**：拿一把**刻度尺**挨个量——每条记录独立打 0–5 分。刻度是绝对的：今天量的 3 分和明天量的 3 分是一个意思。
- **pairwise（成对比较）**：办一场**锦标赛**——批内记录两两过招，LLM 只回答「这两条哪条更好」，赛完用统计模型把胜负战绩折算成排名分。分数是相对的：它说的是「你在这一批里排第几」。

为什么要有锦标赛这种绕弯的玩法？因为 **LLM 判「A 和 B 谁好」比判「A 值几分」稳定得多**——绝对打分容易随措辞漂移（同一条数据换个说法问就 3 变 4），而偏好比较的一致性高。这是 QuRating（ICML 2024）的核心发现，pairwise 因此是默认模式。

两种量具共用同一套**评价准则（rubric）**的不同字段，切换零迁移成本。

## 10.2 rubric：先说清楚"好"是什么

rubric 是一个带**必填标识**的顶层表 + 若干条**准则（criterion）**，每条准则含：

```toml
[rubric]
name = "my-rubric-v1"          # 内联时必填的 rubric 标识（非空字符串），写入 _meta.run.rubric

[[rubric.criteria]]
key = "educational_value"      # 唯一标识，进 _meta.scores
weight = 1.0                   # 聚合权重
description = "教育/训练价值：作为模型训练数据能带来多少可学习的能力。"
pairwise_prompt = "比较两段文本，哪一段更有教育价值、更值得用于训练语言模型？"   # 锦标赛用
pointwise_levels = [ "0: 无学习价值……", …, "5: ……属稀缺的高质量样本。" ]        # 刻度尺用（恰 6 级）
```

内联写进 project.toml 时（`quality.rubric = "inline"`）就是上面这个形态——漏掉 `[rubric]` 的 `name`，启动校验会直接以退出码 2 报「缺失必填键」（完整工程示例见第 7 章）。

系统内置三套（`labelkit rubric --show default:text|default:ui|default:trajectory` 可导出全文，见附录 B）：

- **`default:text`** 四维，直接取自 QuRating：写作水平、事实与知识含量、教育/训练价值、所需专业度；
- **`default:ui`** 四维，面向屏幕数据：截图可读性、树-图一致性（权重 1.5）、状态完整性、交互丰富度；
- **`default:trajectory`** 四维（v1.8），面向操作序列（episode）：完成度、连贯性、目的性、噪声残留——量的是「这段操作是否完整、顺畅、专一、干净」。

`quality.rubric` 不填时按模态自动选默认；v1.8 起若开启了分段算子（stream 模式，第 25 章），不填则**两种模态都**解析为 `default:trajectory`——逐帧视觉判据（`default:ui`）对「无图的整段打分」毫无意义；显式写了选择器时恒以你写的为准。**默认 rubric 衡量的是「作为模型训练数据的价值」**——这个口径不一定是你的口径！第 3 章里那条请假条被淘汰，就是因为它在「事实含量」「专业度」上天然低分。如果你的目标是「保留多样的真实用户请求」，就应该写自己的 rubric（准则设计方法见 10.8 与第 16 章）。

## 10.3 pointwise 模式：绝对刻度尺

**机制**：每条记录 × 每条准则发起一次调用，提示词呈现该准则的 6 级加性量表（0–5，逐级累加式描述——FineWeb-Edu 验证的最优形式），要求模型**先给两句理由、再给整数分**（先理由后结论，缓解「结论先行再找补」的偏差）。得分归一化：除以 5。

```
系统: 按以下 0–5 加性量表为记录的 educational_value 打分，先给两句理由再给整数分：
      0: 无学习价值（噪声、纯广告、无意义重复）。
      1: 学习价值极低，内容浅表。
      ……
      5: 在 4 的基础上，训练价值突出，属稀缺的高质量样本。
用户: [记录内容] 帮我写一条请假条，明天上午要去医院

响应: {"scores": [{"criterion": "educational_value",
       "reason": "意图明确的写作示范任务……但任务简单，可学习内容有限。", "score": 3}]}
→ 归一化 3/5 = 0.6
```

**特性**：分数**跨批、跨运行可比**——0.6 就是 0.6。适合「我要一条全局统一的质量线」「多次运行的产物要合并后统一后筛」的场景。

**成本**：`N × 准则数` 次调用。四条准则的默认 rubric 下，1 万条记录 = 4 万次调用。

## 10.4 pairwise 模式：批内锦标赛

**机制**分四步（默认模式，忠实复刻 QuRating）：

1. **配对**：批内洗牌后相邻配对，重复 `rounds` 轮（默认 k=4）。批内条数为偶数时每条记录恰好参赛 k 次；为奇数时每轮洗牌后落单的那条轮空该轮（参赛次数 ≤ k）。k≥3 时「谁和谁交过手」构成的图高概率连通，保证排名可辨识。配对方案由 `run.seed` 决定，可复现。
2. **裁决**：每对记录发一次调用，A/B 呈现顺序随机（缓解「先出场的占便宜」的位置偏差）。默认一次调用裁决全部准则（`criteria_per_call = "all"`，成本优化；`"single"` 切回 QuRating 原文的每准则独立询问）。输出 `{"judgments": [{"criterion", "winner": "A"|"B"|"tie", "reason"}]}`，经结构引擎校验。
3. **Bradley-Terry 拟合**：把全部胜负喂给 BT 模型——你可以把它理解为**棋手等级分**（Elo 的静态版）：每条记录一个实力值 θ，P(i 胜 j) = θᵢ/(θᵢ+θⱼ)，用极大似然迭代解出最能解释全部战绩的 θ。平局按半胜半负计；每条记录附加微量「对虚拟选手的半胜半负」正则化，保证全胜/全负者的分数依然有限。
4. **归一化**：每条准则独立，把批内 log θ 排名折算成百分位：`score = (rank−1)/(N−1)`，并列的 log θ 取平均秩。**无并列时批内最低为 0、最高为 1**（极端情形如全批裁决均为平局时，所有记录同得 0.5）。

一个可手工核对的四人小锦标赛（单准则、k=2，来自设计规格的走查）：

| 记录 | 战绩（2 场） | log θ | 排名 | score |
|---|---|---|---|---|
| 「哈哈哈哈哈哈」 | 0 胜 0 平 | −3.02 | 1/4 | **0.000** |
| 「帮我写一条请假条…」 | 0 胜 1 平 | −0.08 | 2/4 | **0.333** |
| 「把…翻译成英文」 | 1 胜 1 平 | 0.08 | 3/4 | **0.667** |
| 「解释二分查找为什么 O(log n)…」 | 2 胜 0 平 | 3.02 | 4/4 | **1.000** |

**成本**：每准则组一次调用时为 `N × k / 2` 次（k=4 时 = 2N），**与准则条数无关**——这是它比 pointwise 便宜的原因（四准则下 2N vs 4N）。

### ⚠️ 本章最重要的一段：pairwise 分数是批内相对量

pairwise 的 score 是**百分位**。由此产生三条推论，误解它们是质量门翻车的头号原因：

1. **`threshold = 0.3` 的含义是「淘汰每批相对垫底的约 30%」**，不是「淘汰绝对质量低于某线的记录」。哪怕某一批全是精品，垫底的 30% 照样被淘汰；全是垃圾的批也照样留下 70%。
2. **两批各自的 0.53 不可比**。合并多次运行的产物按分数统一后筛？pairwise 分数干不了这个活，请用 pointwise。
3. **`batch_size` 是质量口径参数**。批越大，百分位越平滑、锦标赛样本越充分；批太小（比如 16）时一两条极端记录就能扭曲全批的百分位。批大小=1 的极端情况：不发起任何比较，直接给 0.5。

**选型口诀**：要**全局统一的绝对质量线**（跨批可比、下游按分数后筛）→ pointwise；要**批内优中选优**（每批固定淘汰掉相对差的）→ pairwise；pairwise + `top_ratio` 是语义最自洽的组合（本来就是按名次选）。

## 10.5 质量门：threshold 与 top_ratio

打分本身不淘汰任何记录；淘汰由**选择机制**执行，二选一（互斥，同时配置直接配置错误）：

| 机制 | 配置 | 行为 | 每批淘汰量 |
|---|---|---|---|
| 分数线 | `selection = "threshold"`（默认）+ `threshold = 0.3` | 聚合分 < 0.3 ⇒ `dropped_lowq` | 随分布浮动 |
| 定量 | `selection = "top_ratio"` + `top_ratio = 0.6` | 批内按聚合分降序**保留前 60%**（ceil(0.6 × 批内已打分存活数) 条） | 恒定比例 |
| 都不设 | `threshold` 缺省 | **只打分不过滤**——分数随 `_meta` 落盘，去留交给下游后筛 | 0 |

聚合分 = Σ(准则分 × 权重) / Σ权重。top_ratio 的并列裁决：聚合分相同时按记录 id 字典序，保证同输入同 seed 完全可复现。

**拿不准阈值时的标准流程**：第一跑不设 threshold → 看报告的 `aggregate_histogram`（10 桶分布）和 rejects → 对着分布画线 → 第二跑再设。直方图上 0.2–0.3 有个 4 条的桶、你的线画 0.25，就意味着这个桶里约一半会被切掉——先知道会切掉什么，再决定切不切。

### 按类分池（v1.7）：每类各办一场锦标赛

开启分类算子（`[classify]`，第 24 章）后，比较池从「批」细分为「批 × 类」：批内存活记录先按类标签分池，配对、BT 拟合、百分位折算与质量门全部在**各自池内**独立进行——写作类只跟写作类比，谁也不给谁当分母；threshold / top_ratio 也按池独立执行（top_ratio 的名额基数 = 池内已打分存活数）。每类还可以在 `[class.X.quality]` 里覆盖自己的 mode / rounds / rubric / threshold / selection / top_ratio（未覆盖的键继承全局；评审团、双序等鲁棒性选项恒为全局）。落盘时 `_meta.scores` 多一个 `pool` 字段自述池归属（仅启用分类时出现）——此时读 pairwise 分数要改口径：0.53 说的不再是「批内百分位」，而是「同类池内百分位」。代价也直白：分池把 10.4 推论 3 的「批太小」问题按类又切了一刀——某类在批内只剩三五条时，锦标赛样本太少、百分位严重失真。两个应对：给小类单独覆盖 `mode = "pointwise"`（绝对刻度不吃池子大小），或增大 `batch_size` 把每个池都喂饱。分池后的账在报告 `quality.by_class` 里逐池给出（有效 mode / rounds、聚合分直方图、每准则均值与 tie 率）。完整语义与配置示例见第 24 章。

### 序列打分（v1.8）：给整段操作打分

开启分段算子（第 25 章）后，被打分的对象从单条记录变成 episode——一次「点外卖」「打个车」的整段操作。打分方式随之改变：

- **纯文本、无图**。提示词呈现 `[步骤序列]`（extract 摘出的动作序列的文字渲染）加 `[成员帧摘要]`（逐帧的有界文字摘要），不带任何截图——所以 stream 下 `quality.llm` 不需要视觉能力（第 4 章约束表的放宽项；v1.9 的缝合判定同样纯文本）。extract 未开启时没有步骤可呈现，退化为纯帧摘要打分，rubric 里的「步骤」读作「帧间变化」（M1 会对这个组合打一条 warning 提示）。
- **fallback 步分列呈现**。extract 摘取失败兜底出的步骤（记为 `other`）与 LLM 确证的 `other` 在步骤序列里**分开标注**——防止工具自己的失败被误读成「操作不连贯」，污染连贯性判据。
- **缝合后打分单元 = 线索**（v1.9，第 26 章）。再开启缝合算子时，episode 碎片在到达打分工位前已被缝成完整线索，评的是跨碎片的整个任务；步骤序列的文字渲染对接缝占位步（`detail.kind == "thread_seam"`）附加专用后缀「（线索接缝：被 X 打断）」，与 fallback 步的分列标注并列——已知的任务中断如实告知评审，防止 trajectory rubric 的噪声残留/连贯性判据把接缝当成噪声或无法解释的跳变扣分。
- **比较池骤然变小**。pairwise 的池子从「批内记录」变成「批内 episode」：一批 256 帧可能只装下十几个 episode，锦标赛样本变少、百分位更易失真（10.4 推论 3 的放大版）。应对：加大 `batch_size`、或改 pointwise；stream 下 `threshold` 缺省=只打分不筛，本来就是合理起点——先看分布再决定筛不筛。
- **长 episode 的分数要降低信任**。超过约 20 步的长序列，LLM 判分信度随长度明显衰减（GUIDE 的实测结论）——对长 episode 建议用 pairwise（相对比较比绝对刻度更稳），或后筛时对其分数放宽容忍、辅以人工抽查。

## 10.6 鲁棒性三件套（默认全关，按需加钱）

| 机制 | 配置 | 原理 | 成本 |
|---|---|---|---|
| **多评审团** | `judges = ["m1","m2","m3"]`（奇数个 profile） | 每次比较各评审独立裁决，按准则多数票。异构小模型评审团优于单一大评审（PoLL 论文），且显著稀释单模型口味偏差 | ×评审数 |
| **双顺序裁决** | `both_orders = true` | 同一对记录正、反两种顺序各裁一次，两次一致才记胜负，不一致记平——把位置偏差从「随机化稀释」升级为「系统性消除」 | ×2 |
| **裁决失败兜底** | `on_unscored = "keep"`（默认）/ `"drop"` | 单次比较修复后仍非法 ⇒ 按平局计（对 BT 中性）；某记录**全部**比较都失败 ⇒ score 置 null，按本开关决定去留。keep 时该记录不占 top_ratio 名额、直接保留 | — |

组合语义：先每评审做双序一致性合成，再跨评审多数票。什么时候上？单模型 + 随机顺序对绝大多数场景够用；当你发现同 seed 重跑的裁决翻转率高（第 16 章的诊断指标）、或者这批数据的取舍直接影响下游重大决策时，再按需加。

## 10.7 让打分留下理由：judgment_reasons

`quality.judgment_reasons`（默认 `"auto"`）控制裁决时是否要求 LLM 附一句话理由：

- `"auto"`：**开了 trace 且订阅了 quality 通道才要理由**——不开 trace 就不要（理由没地方去，纯浪费 token）；
- `true` / `false`：强制开/关。

理由写进 trace 的 `quality.judgment` / `quality.pointwise` 事件，是 rubric 迭代的核心原料（第 16 章的闭环全靠它）。每次裁决约多花 30–60 输出 token；rubric 定稿后的全量运行可以关掉省钱。

## 10.8 rubric 设计指南

**几条准则合适？** 3–5 条。超过 6 条：单次裁决的提示词过长，模型对每条准则的注意力摊薄，且准则间高度相关（都在换着说「质量好」）时纯属浪费。

**先想清楚口径再动笔**：你在为「训练数据价值」打分（默认 rubric 的口径），还是「业务合规性」「信息完整度」「用户意图清晰度」？口径不同，同一条数据的命运完全不同。

**pairwise_prompt 要写成有判据的比较问句**。反例：「哪段文本更好？」（好=什么？）；正例：「哪段文本的诉求更明确、包含更多可执行的约束（时间、对象、格式）？」——判据具体，裁决才稳定。

**pointwise_levels 用加性结构**：每级在上一级基础上**追加**条件（「在 2 的基础上，……」），模型顺着往上爬到爬不动为止，比六个独立描述稳定得多。相邻两级的边界要可判：「基本可读」→「结构清晰」→「行文精炼」是能分的；「不错」→「很好」→「非常好」是不能分的。

**weight 表达你的优先级**，不必都是 1.0。`default:ui` 把树-图一致性设为 1.5 就是明确的价值判断：数据对不上，画质再好也没用。

**准则好不好，用数据说话**：跑小样本、开 trace，看每条准则的 tie 率（>40% ⇒ 区分度不足，改写判据或降权）、准则间胜负相关性（>0.9 ⇒ 冗余，合并）、抽读 reason 找准则没覆盖的评价维度（该加新准则了）。完整的诊断指标表和 jq 命令在第 16 章。

## 10.9 成本速算表

设批大小 B、总记录 N、准则数 C：

| 配置 | 调用次数 |
|---|---|
| pointwise | N × C |
| pairwise（默认 k=4，all-in-one） | N × k/2 = 2N |
| pairwise + judges×3 | 6N |
| pairwise + judges×3 + both_orders | 12N |
| pairwise + criteria_per_call="single" | 2N × C |

再加上 annotate 的 N 次（self-consistency ×n）、verify 的 N 次左右——**quality 几乎总是大头**。预算紧张的排序建议：先 pairwise 默认档（2N）；连 2N 都嫌多时考虑「第一跑 pointwise 单准则粗筛 + 第二跑对存活者精标」的两段式。

## 10.10 结果落在哪

- `_meta.scores`：每准则分 + `__aggregate__` + mode + batch_no（结构见第 8 章）。注意 pairwise 的战绩明细（comparisons/wins/ties/log_theta）只是进程内的中间量，**不落盘**——落盘的只有折算后的百分位分数；
- 报告 `quality` 节：模式、judgment_failures、聚合分直方图、每准则均值；pairwise 另有 `per_criterion_tie_rate`（每准则平局率；只统计拿到裁决的比较，调用级失败不摊进分母）——第 16 章诊断表的 tie 率不再需要 jq，直接读报告；
- 拒绝通道：`stage="quality"`，reason=`below_threshold`（threshold 淘汰）或 `top_ratio`（定量淘汰）；
- trace：`quality.judgment`（每次比较）/ `quality.pointwise`（每次打分）/ `quality.bt_fit`（每批每准则拟合摘要）/ `quality.gate`（每条门控判定）。
